SECTION 1 — PROJECT OVERVIEW
================================================================================

## What this project is
**Tool name: ThreatLens**
GitHub repo: github.com/[yourusername]/ThreatLens

WHAT IS IT?
-----------
A portable, command-line malware analysis tool that accepts a suspicious file
as input and produces a structured threat report. The tool performs static
analysis across multiple file types, enriches findings with external threat
intelligence, scores the sample using a transparent confidence system, and
outputs results in multiple formats.

The tool is designed for:
  - Security students and researchers
  - Junior SOC analysts performing file triage
  - CTF participants
  - Anyone who needs to analyse a suspicious file without uploading it to
    a cloud sandbox

CORE DESIGN PRINCIPLES
-----------------------
  1. Portable — runs on Linux or Windows with minimal setup
  2. Modular — every analysis module is independent and optional
  3. Graceful degradation — missing dependencies skip that module, not crash
  4. Pluggable dynamic backends — static core works standalone; dynamic
     analysis is an optional layer bolted on top
  5. Transparent scoring — every score contribution includes a reason string
  6. Privacy-first — never sends the actual file anywhere; only hashes to VT

DEVELOPMENT ENVIRONMENT
-----------------------
  OS:       Ubuntu Linux (daily driver)
  IDE:      VSCode
  Language: Python 3.x
  AI tools: Claude Pro (chat + Claude Code CLI), GitHub Copilot (auto mode)

SECTION 2 — COMPLETE FEATURE LIST
================================================================================

FOUNDATION LAYER
----------------
  - File intake and validation
  - python-magic for true file type detection (not extension-based)
  - Hash generation: MD5, SHA256, TLSH (fuzzy hash)
  - OS detection for cross-platform binary paths
  - Graceful degradation: missing module = warning + skip, not crash
  - Configurable via config.yaml (paths, API keys, enabled modules)
  - Per-module timeouts to prevent pipeline hangs
  - Parallel module execution where modules are independent

STATIC PE ANALYSIS (Windows Executables)
-----------------------------------------
  Library: pefile
  Extracts:
  - PE headers (machine type, compile timestamp, entry point)
  - Sections (name, size, virtual address, raw size)
  - Per-section entropy (Shannon entropy — flags packed/encrypted sections)
  - Imports and exports (DLL names, function names)
  - Digital signature presence, validity, signer identity
  - Packer/obfuscator detection (UPX, MPRESS, Themida, VMProtect signatures)
  - Suspicious import flagging (VirtualAlloc, WriteProcessMemory, etc.)
  - Resources (language, type, size)

STRING ANALYSIS
---------------
  Libraries: FLOSS, regex
  - FLOSS for deobfuscated string extraction (handles encoded/stacked strings)
  - Raw string extraction pass
  - IOC extractor module (dedicated regex patterns):
      * IPv4 and IPv6 addresses
      * Domain names and URLs
      * File paths (Windows and Linux)
      * Registry keys
      * Mutex names
      * Email addresses
      * Base64 blobs
      * Windows API function names

CAPABILITY DETECTION
--------------------
  Library: capa (Mandiant FLARE team)
  - Detects what the binary is capable of doing
  - Maps capabilities to MITRE ATT&CK TTPs (Tactics, Techniques, Procedures)
  - Example output:
      Capability: HTTP communication
      ATT&CK:     T1071.001 - Application Layer Protocol: Web Protocols
  - Example output:
      Capability: inject into process
      ATT&CK:     T1055 - Process Injection

SIGNATURE MATCHING
------------------
  Library: YARA
  - Matches sample against YARA rule collections
  - Recommended free rulesets to bundle:
      * Malpedia rules (community)
      * ESET public rules
      * Neo23x0 / Florian Roth rules (signature-base on GitHub)
      * Custom rules authored during development
  - Each match contributes to confidence score with rule name as reason

DOCUMENT ANALYSIS
-----------------
  Library: oletools (olevba, mraptor, oleid)
  File types: .doc, .docx, .xls, .xlsx, .ppt, .pptx, .rtf
  - VBA macro extraction and analysis
  - Suspicious macro behaviour detection (auto-execution, shell calls)
  - OLE structure inspection
  - Embedded object detection

  Library: peepdf
  File types: .pdf
  - PDF object and stream inspection
  - Embedded JavaScript extraction
  - Suspicious element detection (launch actions, URI actions)
  - Entropy analysis of streams

THREAT INTELLIGENCE ENRICHMENT
-------------------------------
  API: VirusTotal (free tier — 4 lookups/minute)
  Method: HASH ONLY — the actual file is never uploaded
  Returns:
  - Detection ratio (e.g. 43/72 engines flagged)
  - Known malware family names
  - First submission date
  - Community reputation score
  Note: If no VT API key is configured, this module is silently skipped

CONFIDENCE SCORING SYSTEM
--------------------------
  This is the most important differentiator of the tool.

  Every module returns:
    - data (the raw findings)
    - score_delta (integer, positive or negative)
    - reason (human-readable string explaining the score contribution)

  Final report example:
    ┌─────────────────────────────────────────────┐
    │  THREAT SCORE: 78 / 100   [HIGH RISK]       │
    ├─────────────────────────────────────────────┤
    │  [+20] High entropy in .text section (7.8)  │
    │         → likely packed or encrypted        │
    │  [+25] YARA match: Emotet_dropper_v3        │
    │  [+18] capa: process injection capability   │
    │         ATT&CK T1055                        │
    │  [+15] No valid digital signature found     │
    │  [+10] Suspicious imports: VirtualAlloc,    │
    │         WriteProcessMemory, CreateRemote     │
    │         Thread                              │
    │  [-5]  0/72 VirusTotal detections           │
    │         (hash not previously seen)          │
    │  [-5]  Compile timestamp appears legitimate │
    └─────────────────────────────────────────────┘

  Score bands (suggested):
    0-30:   LOW RISK
    31-55:  MEDIUM RISK
    56-75:  HIGH RISK
    76-100: CRITICAL

OUTPUT FORMATS
--------------
  1. JSON — full structured output, machine-readable, for pipeline integration
  2. Terminal (rich library) — colour-coded, progress bars, formatted tables,
     threat score in colour based on band
  3. HTML report — self-contained, styled with Jinja2 templating, includes:
     - Summary header with filename, hashes, file type, threat score
     - Collapsible sections per module
     - MITRE ATT&CK table with TTP IDs and descriptions
     - IOC table (categorised: IPs, domains, file paths, registry keys)
     - Score breakdown table with reasons
     - YARA matches table

SECTION 3 — DYNAMIC ANALYSIS BACKENDS (OPTIONAL LAYER)
================================================================================

The static core works completely standalone. Dynamic analysis is an optional
layer added after the static pipeline is complete and stable.

PLUGGABLE PROVIDER ARCHITECTURE
---------------------------------
Each dynamic provider implements the same interface:
  - run(sample_path) -> dict (structured behavioural results)
  - is_available() -> bool (checks if provider is configured and reachable)

The main pipeline calls each enabled provider in order and merges results.

BACKEND A — SPEAKEASY (Default Dynamic Option)
-----------------------------------------------
  What it is: Windows binary emulator by Mandiant. Runs PE files, drivers,
  and shellcode in a modelled Windows runtime. Does not require a real Windows
  VM. Runs as a Python library with structured JSON output.

  Why first: Lightweight, Linux-native, safe (no real execution), easy to
  parse and merge into the report.

  What it adds:
  - API call log (what Windows APIs the sample calls)
  - File system activity (what files it tries to create/read/delete)
  - Registry activity
  - Network activity (DNS queries, socket connections)
  - Behaviour-level hints that static analysis cannot see

  Limitations: Some malware detects emulation and changes behaviour.

BACKEND B — Windows VM Worker (Optional, Higher Fidelity)
----------------------------------------------------------
  What it is: A real Windows VM (VMware or VirtualBox) that executes selected
  samples and collects telemetry.

  Setup requirements:
  - VMware Workstation or VirtualBox on Ubuntu host
  - Windows VM with Sysmon installed and configured
  - Snapshot taken BEFORE any sample execution
  - Snapshot reverted AFTER each run (mandatory for repeatable analysis)
  - Disable VMware convenience features: drag-and-drop, copy-paste,
    shared folders — reduces malware evasion triggers

  Telemetry to collect:
  - Sysmon event logs (process creation, network connections, file events,
    registry events, pipe events)
  - Process list before and after
  - Network captures (optional, FakeNet-NG recommended)

  Important: Some malware detects virtualisation and alters behaviour. A VM
  improves fidelity over emulation but does not guarantee full detonation.

  Integration: The VM worker is an optional execution worker, not the default
  path. Only selected samples (e.g. those scoring HIGH or CRITICAL on static)
  should be sent to the VM. The tool calls the VM worker via a local API or
  CLI wrapper script.

BACKEND C — CAPE Sandbox (Advanced, Document Only)
---------------------------------------------------
  What it is: Full open-source malware sandbox with payload extraction,
  config extraction, and rich behavioural reporting. Much heavier than
  Speakeasy, requires a dedicated setup.

  When to add: Only after Speakeasy is working and the static pipeline is
  complete. Treat as a research-grade optional backend, not a core feature.

  For the final year submission: Document CAPE as a supported optional
  backend that was not implemented due to time constraints. This is entirely
  legitimate and shows architectural foresight.

SECTION 4 — PORTABILITY AND DEPLOYMENT
================================================================================

Portability means two things:
  1. The tool can run in different environments without painful setup
  2. The tool degrades gracefully when dependencies are missing

IMPLEMENTATION PRIORITIES (ordered by impact):

1. GRACEFUL DEGRADATION (Low effort, High impact)
   Every module wrapped in try/except. If import fails or binary not found:
   - Log a warning: "FLOSS not found — string deobfuscation skipped"
   - Set module result to None
   - Continue pipeline
   - Note skipped modules in the report

2. pyproject.toml + INSTALL SCRIPT (Low effort, High impact)
   pyproject.toml with all Python dependencies pinned.
   install.sh handles:
   - System packages (libmagic, build tools)
   - Python virtual environment creation
   - pip install of all dependencies
   - Download of default YARA rulesets
   - Verification step confirming tools are available

   User experience target:
     git clone <repo>
     cd malware-triage
     ./install.sh
     python main.py --help

3. CONFIG FILE (Medium effort, High impact)
   config.yaml with:
   - virustotal_api_key: ""
   - yara_rules_dir: "./rules"
   - floss_binary: "./bin/floss"
   - output_dir: "./reports"
   - enabled_modules: [pefile, floss, capa, yara, oletools, peepdf, virustotal]
   - dynamic_provider: "speakeasy"  # or "vm_worker" or "cape" or "none"
   - log_level: "INFO"

   Use pathlib.Path throughout — never hardcode / or \ separators.
   Use tempfile.mkdtemp() — never hardcode /tmp.

4. DOCKER CONTAINER (Medium-High effort, Very High impact)
   Dockerfile that includes:
   - Python environment
   - All system dependencies (libmagic, etc.)
   - FLOSS binary
   - Default YARA rulesets
   - Tool code

   Usage:
     docker build -t malware-triage .
     docker run -v ./samples:/samples malware-triage analyse /samples/file.exe

   This means zero setup for the end user. One command.
   Add docker-compose.yml for convenience.

5. WINDOWS COMPATIBILITY (Low priority, do last)
   - pathlib.Path already handles this if used throughout
   - python-magic needs libmagic DLL on Windows — abstract behind wrapper
   - FLOSS ships Windows and Linux binaries — detect OS and call correct one
   - Test on Windows after everything else works

MODULE INTERFACE STANDARD
--------------------------
Every module returns a dict with this structure:

  {
    "module": "pe_analysis",
    "status": "success",          # or "skipped" or "error"
    "data": { ... },              # raw findings
    "score_delta": 20,            # contribution to confidence score
    "reason": "High entropy in .text section (7.8) suggests packing"
  }

The pipeline collects all module results, sums score_deltas, and passes
everything to the reporter.

------------------------------------------
